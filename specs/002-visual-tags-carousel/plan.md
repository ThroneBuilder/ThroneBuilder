# Implementation Plan: Homepage Visual Redesign — Tags, Filtering & Carousel

**Branch**: `002-visual-tags-carousel` | **Date**: 2026-05-28 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/002-visual-tags-carousel/spec.md`

## Summary

Redesign the ThroneBuilder homepage from a plain article list into a
magazine-style editorial surface. Four user stories are planned together and
implemented one at a time: (1) add tags to articles and a client-side filter bar,
(2) replace the article list with a cover-image grid in varied layouts, (3) add an
auto-rotating carousel above the grid that links to filtered views, and (4) add a
persistent footer with categorized contact and social links on all pages.

All four stories share the existing Astro 5 static site stack. No new frameworks
or build tools are introduced. The implementation is vanilla HTML, CSS, and a
small amount of vanilla JavaScript for the interactive pieces (filtering, carousel
rotation). Each story is independently deployable and checkpointed before the next
begins.

**Structure Decision**: All homepage logic stays in `src/pages/index.astro` as
Astro component-level code and scoped styles. The carousel and filter JS is written
as inline `<script>` tags inside `index.astro` — keeping behavior and markup
co-located for a feature of this scale. If the JS grows beyond ~100 lines per
concern, it moves to `src/utils/`. The footer is in `BaseLayout.astro` — the
only location that guarantees every-page presence.

## Implementation Order (Per User Story)

The user's instruction is "implement and test them one at a time." The dependency
order is:

```
US1 (tags + filter) → US2 (grid + fonts) → US3 (carousel) → US4 (footer)
```

**Why this order**:
- US1 first: adds the `tags` schema field and filter bar. US2 and US3 both read
  article tags, so the schema must exist before the grid or carousel renders tags.
- US2 after US1: the article grid replaces the list and uses the same cards that
  show tags from US1.
- US3 after US1: the carousel's click-to-filter depends on US1's filtering JS
  being in place.
- US4 last: the footer is independent of US1–US3 and can be added at any point,
  but placing it last keeps the earlier stories focused.

**US3 dependency note**: The carousel click behavior calls the same filter
function defined in US1. US1 must be complete and tested before implementing the
carousel click handler.

## Key Technical Decisions (Summary)

| Decision | Choice | Tradeoff Summary |
|----------|--------|-----------------|
| Tag filtering | Vanilla JS + data attributes | Zero dependencies; ~20 lines; no-JS fallback shows all articles |
| Carousel | Vanilla JS + CSS transitions | Zero dependencies; ~60 lines; no-JS shows horizontal scroll strip |
| Display font | Playfair Display (Google Fonts) | ~30 KB; editorial serif matching Salon aesthetic; Aptos Black not a web font |
| Article grid | CSS Grid + :nth-child variants | No JS required; three card types (hero, standard, text-only) |
| Carousel config | `src/data/carousel.ts` TypeScript array | Type-safe; single file; separates config from template |
| No-JS strategy | Inline script adds `js` class to `<html>` | FOUC-free; standard progressive enhancement; 2 lines |
| Footer placement | `BaseLayout.astro` | Single location for all-pages chrome |
