# Implementation Plan: ThroneBuilder.com Site Creation

**Branch**: `thronebuilder-creation` | **Date**: 2026-05-26 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/thronebuilder-creation/spec.md`

## Summary

Build ThroneBuilder.com as a static Astro 5 site with Markdown-based content
stored in GitHub, deployed to Render via auto-deploy on push to `main`. The site
consolidates ThroneBuilder content (articles + embedded YouTube videos) under the
owned domain with full SEO output (sitemap, RSS, OpenGraph). All four user stories
(content browsing, video embedding, SEO/discovery, custom domain) are independently
deliverable and tested before proceeding to the next.

## Technical Context

**Language/Version**: TypeScript 5.x / Node.js 22 LTS

**Primary Dependencies**:
- `astro` (5.x latest stable) — site framework
- `@astrojs/sitemap` — auto-generate XML sitemap at build time
- `@astrojs/rss` — RSS 2.0 feed endpoint
- `zod` (bundled with Astro) — content schema validation

**Storage**: Markdown files in `src/content/blog/` (git-tracked). No database.
Site config constants in `src/consts.ts`.

**Testing**: Manual Lighthouse audit in Chrome DevTools; browser-based
acceptance testing per user story checkpoint. No automated test suite for v1
(per constitution: incremental, test-gated, not test-first for content sites).

**Target Platform**: Static HTML/CSS served from Render CDN; browsers (Chrome,
Firefox, Safari, desktop and mobile)

**Project Type**: Static web site (Astro `output: 'static'`)

**Performance Goals**: Lighthouse ≥ 90 across Performance, Accessibility,
Best Practices, SEO on all pages (Constitution Principle V)

**Constraints**:
- Core content readable without JavaScript (Constitution Principle V)
- WCAG 2.1 AA accessibility
- No SSR — build-time only
- No CSS framework — Astro scoped styles + global CSS custom properties
- Secrets/env vars via Render environment (no hardcoded values)

**Scale/Scope**: Single author, small article volume (<100 articles for v1),
no concurrent write concerns. All complexity is build-time.

**Structure Decision**: Single-project Astro layout following the framework's
own conventions (`src/content/`, `src/pages/`, `src/layouts/`). No
`frontend/`/`backend/` split — there is no backend. No `tests/` directory for
v1 — testing is manual browser-based per constitution guidance for this project
type.
